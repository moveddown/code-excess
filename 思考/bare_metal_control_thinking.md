# 裸机系统中保证 PID 稳定的思考

## 这份材料想解决什么问题？

这份思考材料专门对应下面这个真实场景：

- 没有 RTOS
- 有 PID 控制
- 有报警任务
- 有 4G/HTTP 上传
- MCU 性能有限，不能靠多线程硬拆

你前面的理解已经非常接近正确方向了，核心思想可以整理成一句话：

`用统一时基管理节拍，用非阻塞方式推进任务，用高低优先级分层保护 PID 的稳定性。`

## 优化后的核心理解

你原来的理解可以再精炼成下面 6 条：

1. 系统需要一个统一时基，比如 `1ms tick`
2. 周期性任务按各自周期被释放，不是由主循环“走到哪里算哪里”
3. 事件性任务由中断先记录事件，主循环再消费事件
4. 每个后台任务每次只做一小步，做完立刻退出
5. PID 是固定周期任务，不建议写成很长的流程状态机
6. 保护动作和报警上报不是一回事，必须分开

这 6 条里面，最关键的是第 2 条和第 4 条。

## 不同任务应该怎么分类？

在这种系统里，建议把任务分成 4 层。

### 1. 保护层

这层处理的是必须立刻动作的事情，比如：

- 过流立即关输出
- 过温立即切断功率
- 传感器失效后的紧急保护

这类逻辑不能等主循环慢慢轮询，应该尽量放在：

- 硬件比较器
- 快速中断
- 定时中断中的极短判断

注意：这里是`快速动作`，不是`完整流程处理`。

### 2. 控制层

这层就是 PID、采样、输出刷新。

特点是：

- 周期固定
- 优先级高
- 执行时间要短且稳定

如果控制周期是 `10ms`，那就应该让控制任务严格按 `10ms` 被释放。

### 3. 管理层

这层处理系统内部的业务判断，比如：

- 报警条件统计
- 报警灯闪烁节奏
- 故障锁存
- 状态切换

这层也重要，但通常优先级低于控制层。

### 4. 通信层

这层处理：

- 4G AT 指令
- HTTP 上传
- 日志上报
- 参数同步

这层最容易写成阻塞式，也是最容易把 PID 拖垮的一层，所以一定要做成非阻塞状态机。

## 统一时基到底意味着什么？

统一时基不是说“所有任务都 1ms 执行一次”，而是：

- 系统有一个公共时间参考，比如 `sys_tick_ms`
- 每个周期任务根据自己的 `period_ms` 决定什么时候到期
- 到期后任务被标记为 ready
- 主循环执行 ready 的任务

所以真正的逻辑是：

- 中断负责报时
- 主循环负责执行

而不是主循环自己决定时间。

## 周期任务和事件任务要分开看

很多人刚开始会把所有任务都理解成“每隔多少毫秒检查一次”，但其实系统里至少有两类任务。

### 周期任务

典型例子：

- PID 每 10ms 一次
- 报警扫描每 20ms 一次
- LED 节拍每 100ms 一次

这类任务主要靠时间驱动。

### 事件任务

典型例子：

- UART 收到一帧数据
- ADC 转换完成
- 某个故障中断到来
- 4G 模组返回 `OK`

这类任务主要靠事件驱动。

实际工程里，这两类通常是混合使用的。

比如 4G 上传就是：

- 用周期任务推进状态机
- 用串口接收事件唤醒解析

## PID 为什么不能像 4G 一样拆很多阶段？

这个点很重要。

4G/HTTP 是典型流程型任务，所以适合拆成：

- 发送命令
- 等待响应
- 解析结果
- 进入下一状态

但 PID 通常不是这种任务。

PID 更像是：

1. 到周期了
2. 读取当前反馈
3. 计算控制输出
4. 写回 PWM/DAC
5. 结束

所以：

- 4G 更像流程状态机
- PID 更像固定周期计算任务

不要把 PID 想成“分 3 次、4 次才能算完”的东西。除非控制算法特别复杂，否则一次周期内就应该完成。

## 什么才是好的主循环？

好的裸机主循环不是“所有东西都放进 while(1) 里轮流跑”，而是：

- 只执行已经到期或者已经就绪的任务
- 每个任务一次只做小步
- 顺序体现优先级

比如：

```c
while (1) {
    if (pid_due) {
        pid_task_step();
    }

    if (alarm_due) {
        alarm_task_step();
    }

    if (modem_due || uart_event) {
        modem_task_step();
    }
}
```

这个顺序其实已经表达出优先级了：

- PID 优先级最高
- 报警其次
- 通信最低

## 4G/HTTP 为什么必须非阻塞？

因为通信天然慢，而且慢得不稳定。

最危险的写法是这种：

```c
send_http_cmd();
while (!recv_ok()) {
}
```

或者：

```c
send_http_cmd();
delay_ms(2000);
check_result();
```

这类写法会导致：

- PID 周期被拖长
- 报警扫描抖动
- 串口解析变得混乱

正确做法是把通信拆成小步骤：

1. 发送一条命令
2. 退出
3. 串口中断收数据
4. 下次主循环再看是否收到响应
5. 收到后再进入下一状态

这样即使一次上传总共花 3 秒，CPU 也没有被卡死 3 秒。

## 保护和报警为什么必须分开？

因为它们的实时性完全不同。

### 保护

关注的是：

- 立刻停机
- 立刻关输出
- 立刻切断驱动

### 报警

关注的是：

- 蜂鸣器怎么叫
- 指示灯怎么闪
- 要不要上报 4G
- 是否记录日志

保护是立即动作，报警是后续表现。

最容易出问题的写法是把这两件事绑在同一个“报警任务”里，导致真正该快速处理的保护被拖慢。

## 一个更完整的优化模型

把你的思路优化后，可以变成下面这个模型：

### 中断里做什么

- 维护 `sys_tick_ms`
- 锁存采样值
- 执行极快的保护判断
- 释放周期任务
- 记录串口接收事件

### 主循环里做什么

- 先跑 PID
- 再跑报警管理
- 最后跑 4G 状态机
- 所有后台任务都必须快速返回

## 任务释放不等于任务必须补做

这个点非常关键，也是很多人一开始没想到的。

比如 PID 周期是 `10ms`，结果因为某些原因，主循环 35ms 没来得及执行 PID。

这时你有两个选择：

- 补算 3 次 PID
- 只执行最新的一次，并记录自己超期了

在很多控制系统里，第二种更合理。

因为补算历史周期并不能真的把时间倒回去，反而可能让控制器的内部状态更乱。

所以更实用的策略往往是：

- 周期任务只保留一个 ready
- 如果错过新的释放点，就记录 `missed_deadlines`
- 控制任务只做最新的一次

这也是为什么示例代码里会给周期任务加 `missed_releases` 计数。

## 你原来的理解，哪些是对的？

下面这些理解是对的：

- 要有统一心跳
- 各任务有自己的周期
- 每次任务只做一小步
- 执行完就退出，等下次再推进
- 这样看起来多个任务就像同时在工作

## 你原来的理解，还需要补充什么？

主要补 4 点：

1. 不是所有任务都要拆成状态机，PID 更像固定周期计算任务
2. 系统里除了周期触发，还有事件触发
3. 保护动作不能等同于报警处理
4. 需要考虑任务超期和丢周期的问题

## 推荐的工程实现习惯

如果你后面要把这个思路真正带到项目里，建议你养成下面这些习惯：

1. 所有任务都写清楚周期
2. 所有任务都写清楚最大执行时间
3. 所有耗时流程都禁止 `delay` 和 `while` 死等
4. 所有通信都改成状态机
5. 所有紧急保护都单独走快速通道
6. 对控制任务增加超期统计

## 这份示例代码对应什么目的？

同目录下的 `bare_metal_scheduler_example.c` 不是为了直接上板，而是为了把这几个概念放到同一份代码里看清楚：

- 统一 tick
- 周期任务释放
- 事件驱动的串口响应
- PID 固定周期执行
- 报警任务单独运行
- 4G 状态机非阻塞推进
- 周期超期计数

你可以把它当作一份“架构思路样板”。

## 一句话总结

把你的想法再优化一句，就是：

`不是让所有任务都抢着跑，而是让控制任务有稳定节拍，让保护有快速通道，让通信永远只向前走一小步。`

这套思路对于没有 RTOS 的项目尤其重要。
